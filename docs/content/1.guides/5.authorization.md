Authorization in Ginjou is separate from authentication. Authentication tells you who the user is. Authorization tells you what that user can do.

## Mental Model

Keep authorization decisions close to the UI surface they affect.

1. Authentication establishes who the user is.
2. The authz provider decides what that user can do.
3. `useCanAccess` guards one action or one view.
4. `usePermissions` exposes the broader permission model.

That separation keeps the rules explicit. A page can be authenticated and still restrict specific actions.

## Authz Contract

```ts
interface Authz {
	access?: (params: AccessCanParams) => Promise<AccessCanResult>
	getPermissions?: (params?: any) => Promise<GetPermissionsResult>
}
```

Both methods are optional so you can start with the part your application actually needs.

## Setup

Register the provider at the root.

::code-group

```vue [Vue]
<script setup lang="ts">
import { defineAuthzContext } from '@ginjou/vue'

defineAuthzContext({
	async access({ resource, action, params }) {
		if (resource === 'posts' && action === 'edit') {
			return { can: params?.ownerId === 'me' }
		}

		return { can: false }
	},
	async getPermissions() {
		return ['posts:read', 'posts:edit']
	},
})
</script>
```

## useCanAccess

Use `useCanAccess` when the UI needs to decide whether an action should be available.

### Composition

- Action: `authz.access`
- Query key: authz access state plus `resource`, `action`, `params`, and `meta`
- Query enabled rule: runs only when the provider exposes `access`, and respects `queryOptions.enabled`
- Query callbacks: forwards success and error callbacks from `queryOptions`
- Output: an access result that must include `can`

That result can also carry extra information if your application wants to show a reason or log the decision, but the essential part is still `can`.

### Example

This pattern works best when the permission check lives next to the action itself. That makes the condition easier to audit later.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useCanAccess } from '@ginjou/vue'

const { data: canEdit } = useCanAccess({
	resource: 'posts',
	action: 'edit',
	params: { id: '42', ownerId: 'me' },
})
</script>

<template>
	<button v-if="canEdit?.can">
		Edit
	</button>
</template>
```

## usePermissions

Use `usePermissions` when the UI needs the raw permission model.

This is useful for shell-level decisions, role-driven menus, or complex page layouts where many permission checks share the same upstream permission set.

### Composition

- Action: `authz.getPermissions`
- Query key: authz permissions state plus optional params
- Query enabled rule: runs only when the provider exposes `getPermissions`, and respects `queryOptions.enabled`
- Query callbacks: forwards success and error callbacks from `queryOptions`

Use this when the application already has one upstream permission model and many UI decisions should reuse it.

### Example

::code-group

```vue [Vue]
<script setup lang="ts">
import { computed } from 'vue'
import { usePermissions } from '@ginjou/vue'

const { data: permissions } = usePermissions<string[]>()

const canManagePosts = computed(() => {
	const values = permissions.value ?? []
	return values.includes('posts:edit')
})
</script>

<template>
	<button v-if="canManagePosts">
		Manage posts
	</button>
</template>
```

## Role-Based Access

```ts
defineAuthzContext({
	async access({ action }) {
		const role = 'editor'
		const permissionsByRole = {
			admin: ['create', 'read', 'edit', 'delete'],
			editor: ['create', 'read', 'edit'],
			viewer: ['read'],
		}

		return {
			can: permissionsByRole[role as keyof typeof permissionsByRole].includes(action),
		}
	},
})
```

This shape works well when the backend or app session already exposes one stable role such as `admin`, `editor`, or `viewer`.

## Resource Ownership

```ts
defineAuthzContext({
	async access({ resource, action, params }) {
		if (resource === 'posts' && action === 'edit') {
			return { can: params?.ownerId === params?.currentUserId }
		}

		return { can: false }
	},
})
```

Use this when access depends on the current record rather than a global role.

## Permission-Driven Navigation

When menus or layout sections depend on one shared permission list, fetch that list once and derive visible UI from it.

::code-group

```vue [Vue]
<script setup lang="ts">
import { computed } from 'vue'
import { usePermissions } from '@ginjou/vue'

const { data: permissions } = usePermissions<string[]>()

const menuItems = computed(() => {
	const values = permissions.value ?? []

	return [
		{ label: 'Posts', to: '/posts', visible: values.includes('posts:read') },
		{ label: 'Users', to: '/users', visible: values.includes('users:read') },
		{ label: 'Settings', to: '/settings', visible: values.includes('settings:read') },
	].filter(item => item.visible)
})
</script>

<template>
	<nav>
		<a
			v-for="item in menuItems"
			:key="item.to"
			:href="item.to"
		>
			{{ item.label }}
		</a>
	</nav>
</template>
```

## Practical Guidance

- Use `useCanAccess` close to the button, link, or action surface.
- Use `usePermissions` when you need broader permission state that many UI decisions can share.
- Keep authorization checks explicit. Hidden UI is useful, but the backend must still enforce the rule.
- Prefer simple permission shapes first. Add rich metadata only when the UI truly needs to explain decisions.
