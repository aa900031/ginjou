Authorization in Ginjou is separate from authentication. Authentication answers who the user is. Authorization answers what that user can do.

## Mental Model

Keep authorization decisions close to the UI surface they affect.

1. Authentication establishes the current user.
2. The authz provider evaluates access decisions.
3. `useCanAccess` answers one permission check.
4. `usePermissions` loads a broader permission model that many UI decisions can reuse.

That separation keeps the rules explicit. A page can be authenticated and still restrict specific actions.

## Authz Contract

```ts
interface AccessCanParams {
	action: string
	resource?: string
	params?: Record<string, unknown>
	meta?: Record<string, unknown>
}

interface AccessCanResult {
	can: boolean
	reason?: string
}

interface Authz {
	access?: (params: AccessCanParams) => Promise<AccessCanResult>
	getPermissions?: (params?: any) => Promise<GetPermissionsResult>
}
```

Both methods are optional so you can start with the part your application actually needs.

## Setup

Register the provider at the root.

::code-group

```vue [Vue]
<script setup lang="ts">
import { defineAuthzContext } from '@ginjou/vue'

defineAuthzContext({
	async access({ resource, action, params }) {
		if (resource === 'orders' && action === 'show') {
			return { can: params?.customerId === params?.currentUserId }
		}

		return { can: false, reason: 'Not allowed for the current role' }
	},
	async getPermissions() {
		return ['orders:read', 'shipments:update']
	},
})
</script>
```

::

## useCanAccess

Use `useCanAccess` when the UI needs to decide whether an action should be available.

### Composition

- Action: `authz.access`
- Query key: authz access state plus `resource`, `action`, `params`, and `meta`
- Query enabled rule: runs only when the provider exposes `access`, and respects `queryOptions.enabled`
- Query callbacks: forwards success and error callbacks from `queryOptions`
- Output: an access result that must include `can`

That result can also carry extra information such as a reason string that the UI may surface.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useCanAccess } from '@ginjou/vue'

const { data: canEdit } = useCanAccess({
	resource: 'orders',
	action: 'edit',
	params: {
		id: 'order-42',
		ownerId: 'user-1',
		currentUserId: 'user-1',
	},
})
</script>

<template>
	<button v-if="canEdit?.can">
		Edit order
	</button>
	<p v-else>
		{{ canEdit?.reason ?? 'You cannot edit this order.' }}
	</p>
</template>
```

::

Because `useCanAccess` is query-backed, repeated checks for the same decision can be cached instead of recalculated on every render.

## usePermissions

Use `usePermissions` when the UI needs the raw permission model.

This is useful for shell-level decisions, role-driven menus, or complex page layouts where many permission checks share the same upstream permission set.

::code-group

```vue [Vue]
<script setup lang="ts">
import { usePermissions } from '@ginjou/vue'
import { computed } from 'vue'

const { data: permissions } = usePermissions<string[]>()

const canManageShipments = computed(() => {
	const values = permissions.value ?? []
	return values.includes('shipments:update')
})
</script>

<template>
	<button v-if="canManageShipments">
		Update shipment status
	</button>
</template>
```

::

Use this when many permission checks depend on the same upstream dataset. It avoids repeated per-button calls when the application already has a compact permission model.

## RBAC For An E-commerce Back Office

Role-based access control works well when the application exposes a stable set of roles.

| Role | Products | Orders | Shipments |
| :--- | :--- | :--- | :--- |
| `admin` | Full access | Full access | Full access |
| `merchandiser` | Create, edit, publish | Read only | No access |
| `support` | Read only | Read and update status | No access |
| `warehouse` | No access | Read only | Update shipment status |

```ts
defineAuthzContext({
	async access({ resource, action, params }) {
		const role = params?.role

		const matrix = {
			admin: ['products:create', 'products:edit', 'orders:edit', 'shipments:edit'],
			merchandiser: ['products:create', 'products:edit'],
			support: ['orders:show', 'orders:edit'],
			warehouse: ['shipments:show', 'shipments:edit'],
		}

		return {
			can: matrix[role as keyof typeof matrix]?.includes(`${resource}:${action}`) ?? false,
		}
	},
})
```

## ABAC And Record-Level Rules

Attribute-based access control is a better fit when the decision depends on record data, tenant scope, ownership, or department rules.

```ts
defineAuthzContext({
	async access({ resource, action, params }) {
		if (resource === 'orders' && action === 'show') {
			return { can: params?.customerId === params?.currentUserId }
		}

		if (resource === 'shipments' && action === 'edit') {
			return { can: params?.warehouseId === params?.currentWarehouseId }
		}

		return { can: false }
	},
})
```

Use this when access depends on the current record rather than a global role.

## Passing Identity And Record Context

Record-level checks usually need both the current identity and some record-specific context.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useCanAccess, useGetIdentity, useGetOne } from '@ginjou/vue'
import { computed } from 'vue'

const order = useGetOne({
	resource: 'orders',
	id: 'order-42',
})

const { data: identity } = useGetIdentity<{ id: string, warehouseId?: string }>()

const accessParams = computed(() => ({
	currentUserId: identity.value?.id,
	currentWarehouseId: identity.value?.warehouseId,
	customerId: order.record.value?.customer_id,
	warehouseId: order.record.value?.warehouse_id,
}))

const { data: canEdit } = useCanAccess({
	resource: 'orders',
	action: 'edit',
	params: accessParams,
})
</script>
```

::

This keeps the rule explicit: authentication loads who the user is, the page loads the record data it already needs, and authorization receives both as input.

## Integrating External Authorization Engines

Ginjou does not ship with built-in adapters for authorization engines. The contract is intentionally small so you can wrap the engine that already fits your product.

- Casbin is a good fit for policy-driven RBAC or ABAC.
- CASL is a good fit when you want declarative client-side abilities.
- Cerbos is a good fit when authorization is delegated to an external policy service.
- AccessControl.js is a good fit for smaller role hierarchies.

```ts
defineAuthzContext({
	async access(params) {
		const can = await cerbos.isAllowed({
			principal: params.params?.currentUserId,
			resource: params.resource,
			action: params.action,
			context: params.params,
		})

		return { can }
	},
})
```

The important promise here is compatibility, not a built-in integration package.

## UI Checks Versus Backend Enforcement

Authorization in Ginjou improves user experience. It does not replace backend enforcement.

- UI checks hide or disable actions that the user should not attempt.
- Backend checks remain responsible for real security.
- Client-side permission state can go stale, so destructive operations must stay protected at the API layer.

## Permission-Driven Navigation

When menus or layout sections depend on one shared permission list, fetch that list once and derive visible UI from it.

::code-group

```vue [Vue]
<script setup lang="ts">
import { usePermissions } from '@ginjou/vue'
import { computed } from 'vue'

const { data: permissions } = usePermissions<string[]>()

const menuItems = computed(() => {
	const values = permissions.value ?? []

	return [
		{ label: 'Products', to: '/admin/products', visible: values.includes('products:read') },
		{ label: 'Orders', to: '/admin/orders', visible: values.includes('orders:read') },
		{ label: 'Shipments', to: '/admin/shipments', visible: values.includes('shipments:read') },
	].filter(item => item.visible)
})
</script>

<template>
	<nav>
		<a
			v-for="item in menuItems"
			:key="item.to"
			:href="item.to"
		>
			{{ item.label }}
		</a>
	</nav>
</template>
```

::

This avoids repeating the same permission lookup for every single navigation item.

## Practical Guidance

- Use `useCanAccess` close to the button, link, or action surface.
- Use `usePermissions` when many UI decisions can share one permission query.
- Keep authorization checks explicit. Hidden UI is useful, but the backend must still enforce the rule.
- Prefer simple permission shapes first. Add richer metadata only when the UI truly needs to explain decisions.
