Notifications are provider-driven, just like fetchers and auth. Ginjou does not prescribe a UI library. It only expects a small contract for opening and closing messages.

## Mental Model

Notifications are a shared UI capability, not page-local state.

1. You register one notification provider near the root.
2. Components and composables call `useNotify()`.
3. The provider decides how feedback is rendered.
4. Other Ginjou features can reuse the same feedback surface.

That separation is what makes notifications useful for auth flows, undoable mutations, and custom actions without coupling the app to one toast library.

## Notification Contract

```ts
interface Notification {
	open: (params: OpenNotificationParams) => void
	close: (key: string) => void
}
```

## Setup

Register the notification provider near the root of your app.

::code-group

```vue [Vue]
<script setup lang="ts">
import { defineNotificationContext } from '@ginjou/vue'

defineNotificationContext({
	open(params) {
		console.log('open notification', params)
	},
	close(key) {
		console.log('close notification', key)
	},
})
</script>
```

## A Practical Pattern

Treat notifications as the final step of an action flow.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useCreateOne, useNotify } from '@ginjou/vue'

const notify = useNotify()
const createPost = useCreateOne({ resource: 'posts' })

async function submit() {
	try {
		await createPost.mutateAsync({
			params: {
				title: 'New post',
				status: 'draft',
			},
		})

		notify({
			type: 'success',
			message: 'Post created',
			description: 'You can continue editing from the list page.',
		})
	}
	catch {
		notify({
			type: 'error',
			message: 'Could not create the post',
			description: 'Check the form values and try again.',
		})
	}
}
</script>
```

## useNotify

`useNotify` is the main helper for triggering feedback from components, composables, or action handlers.

### Composition

- Action: `notification.open`
- Returned value: a notify function created from the current notification context
- Usage model: can be called directly from components, composables, and custom action handlers

Use it when the feature wants to surface feedback without depending on a specific toast library.

### Example

::code-group

```vue [Vue]
<script setup lang="ts">
import { useNotify } from '@ginjou/vue'

const notify = useNotify()

function saveSucceeded() {
	notify({
		type: 'success',
		message: 'Post saved',
		description: 'The changes are now available.',
	})
}
</script>
```

### Props

Every notification starts from one shared shape.

- `type`: the notification variant
- `message`: the main user-facing message
- `description`: optional supporting detail
- `key`: optional stable identifier for replacing, deduplicating, or closing a specific notification

The built-in notification types are:

- `success`
- `error`
- `progress`

### `success` and `error`

`success` and `error` use the shared props only.

- Use `success` when an action finished in a way the user should know about.
- Use `error` when the action failed and the user may need to retry or change input.
- `description` is useful when the short title is not enough on its own.

These are the most common notification types for custom actions and page-level handlers.

### `progress`

`progress` adds three required props on top of the shared shape.

- `timeout`: how long the undo window stays open
- `onFinish`: what should happen when the countdown completes
- `onCancel`: what should happen when the user cancels during that window

In Ginjou, `progress` is mainly used for undoable update and delete flows rather than upload progress. Built-in update and delete mutations create this notification shape when `mutationMode` is `undoable`, which is the pattern behind undo-style edit and delete experiences.

That means `progress` is best understood as a reversible action notification: the user gets a short time window to change their mind before the mutation is finalized.

::code-group

```vue [Vue]
<script setup lang="ts">
import { MutationMode } from '@ginjou/core'
import { useUpdateOne } from '@ginjou/vue'

const updatePost = useUpdateOne({
	resource: 'posts',
	mutationMode: MutationMode.Undoable,
	undoableTimeout: 5000,
})

function publishPost() {
	updatePost.mutate({
		id: '42',
		params: {
			status: 'published',
		},
	})
}
</script>
```

### Component-Level Example

::code-group

```vue [Vue]

import { useNotify } from '@ginjou/vue'

const notify = useNotify()

function archiveSucceeded() {
	notify({
		type: 'success',
		message: 'Post archived',
		description: 'The item is now hidden from the default list.',
	})
}
</script>
```

## Practical Guidance

- Use success and error notifications for user-visible action outcomes.
- Use progress for undoable flows that need a short reversal window.
- Keep notification text short and specific to the action that just happened.
- Let the provider own presentation, and let the feature own meaning.
