title: Notifications

Notifications are provider-driven, just like fetchers and auth. Ginjou does not prescribe a UI library. It only expects a small contract for opening and closing messages.

## Mental Model

Notifications are a shared UI capability, not page-local state.

1. You register one notification provider near the root.
2. Components and composables call `useNotify()`.
3. The provider decides how feedback is rendered.
4. Other Ginjou features can reuse the same feedback surface.

That separation is what makes notifications useful for auth flows, undoable mutations, and custom actions without coupling the app to one toast library.

## Notification Contract

```ts
interface Notification {
	open: (params: OpenNotificationParams) => void
	close: (key: string) => void
}
```

The built-in notification types used by Ginjou are `success`, `error`, and `progress`.

## Setup

Register the notification provider near the root of your app.

::code-group

```vue [Vue]
<script setup lang="ts">
import { defineNotificationContext } from '@ginjou/vue'

defineNotificationContext({
	open(params) {
		console.log('open notification', params)
	},
	close(key) {
		console.log('close notification', key)
	},
})
</script>
```

::

The provider owns rendering details such as placement, icons, colors, dismiss behavior, and progress UI. Features should only decide when to notify and what the message means.

## useNotify

`useNotify` is the main helper for triggering feedback from components, composables, or action handlers.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useNotify } from '@ginjou/vue'

const notify = useNotify()

function saveSucceeded() {
	notify({
		type: 'success',
		message: 'Post saved',
		description: 'The changes are now available.',
	})
}
</script>
```

::

Use `useNotify` for application-specific events such as custom workflow steps, manual retries, or UI-only feedback that does not belong in a fetcher or controller.

## Let Data Flows Emit System Messages

Most CRUD screens should not build toast text inline in every click handler.

- built-in data hooks accept `successNotify` and `errorNotify` options when the page needs custom copy
- controllers inherit the same notification surface because they are built on top of the data layer
- the provider keeps the final rendering consistent across create, update, delete, login, logout, and undo flows

That keeps message wording centralized instead of scattering toast strings across many pages.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useCreateOne } from '@ginjou/vue'

const createPost = useCreateOne({
	resource: 'posts',
	successNotify: {
		message: 'Post created',
		description: 'You can continue editing from the list page.',
	},
})
</script>
```

::

## Avoid Duplicate Feedback

Notifications are most useful when they add information that the current screen does not already show.

- Use a toast for a successful background save or a route transition that hides the original form.
- Prefer inline field errors when the problem is local to one form input.
- Avoid showing the same failure both as a banner and as a toast unless the flow really needs both.

## Undoable Flows Use Progress Notifications

`progress` adds three required props on top of the shared shape.

- `timeout`: how long the undo window stays open
- `onFinish`: what should happen when the countdown completes
- `onCancel`: what should happen when the user cancels during that window

In Ginjou, `progress` is mainly used for undoable update and delete flows rather than upload progress.

::code-group

```vue [Vue]
<script setup lang="ts">
import { MutationMode } from '@ginjou/core'
import { useUpdateOne } from '@ginjou/vue'

const updatePost = useUpdateOne({
	resource: 'posts',
	mutationMode: MutationMode.Undoable,
	undoableTimeout: 5000,
})

function publishPost() {
	updatePost.mutate({
		id: '42',
		params: {
			status: 'published',
		},
	})
}
</script>
```

::

For `progress`, the provider must support `timeout`, `onFinish`, and `onCancel`.

## Translate System Messages, Not Just App Copy

Notifications are one of the main places where Ginjou consumes i18n.

- resource-aware success messages can use translated resource labels
- error messages can share one consistent translation key set
- undoable progress text can stay aligned with the rest of the system language

Read [I18n](/guides/i18n) when the notification layer should use translated system messages.
