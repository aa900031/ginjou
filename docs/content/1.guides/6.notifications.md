Notifications are provider-driven, just like fetchers and auth. Ginjou does not prescribe a UI library. It only expects a small contract for opening and closing messages.

## Mental Model

Notifications are a shared UI capability, not page-local state.

1. You register one notification provider near the root.
2. Components and composables call `useNotify()`.
3. The provider decides how feedback is rendered.
4. Other Ginjou features can reuse the same feedback surface.

That separation is what makes notifications useful for auth flows, undoable mutations, upload progress, and custom actions without coupling the app to one toast library.

## Notification Contract

```ts
interface Notification {
	open: (params: OpenNotificationParams) => void
	close: (key: string) => void
}
```

## Setup

Register the notification provider near the root of your app.

::code-group

```vue [Vue]
<script setup lang="ts">
import { defineNotificationContext } from '@ginjou/vue'

defineNotificationContext({
	open(params) {
		console.log('open notification', params)
	},
	close(key) {
		console.log('close notification', key)
	},
})
</script>
```

## A Practical Pattern

Treat notifications as the final step of an action flow.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useCreateOne, useNotify } from '@ginjou/vue'

const notify = useNotify()
const createPost = useCreateOne({ resource: 'posts' })

async function submit() {
	try {
		await createPost.mutateAsync({
			params: {
				title: 'New post',
				status: 'draft',
			},
		})

		notify({
			type: 'success',
			message: 'Post created',
			description: 'You can continue editing from the list page.',
		})
	}
	catch {
		notify({
			type: 'error',
			message: 'Could not create the post',
			description: 'Check the form values and try again.',
		})
	}
}
</script>
```

## useNotify

`useNotify` is the main helper for triggering feedback from components, composables, or action handlers.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useNotify } from '@ginjou/vue'

const notify = useNotify()

function saveSucceeded() {
	notify({
		type: 'success',
		message: 'Post saved',
		description: 'The changes are now available.',
	})
}
</script>
```

## Progress Notifications

Progress notifications exist for flows that are long-running or cancelable.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useNotify } from '@ginjou/vue'

const notify = useNotify()

function startUpload() {
	notify({
		type: 'progress',
		message: 'Uploading assets',
		timeout: 5000,
		onFinish: () => console.log('upload finished'),
		onCancel: () => console.log('upload cancelled'),
	})
}
</script>
```

For `progress`, the provider must support:

- `timeout`
- `onFinish`
- `onCancel`

If your UI library does not have a built-in progress toast, you can still implement the contract with a small custom component.

### Upload Example

::code-group

```vue [Vue]
<script setup lang="ts">
import { ref } from 'vue'
import { useNotify } from '@ginjou/vue'

const notify = useNotify()
const isUploading = ref(false)

function startUpload() {
	isUploading.value = true

	notify({
		type: 'progress',
		message: 'Uploading assets',
		description: 'Large files may take a few seconds.',
		timeout: 5000,
		onFinish: () => {
			isUploading.value = false
		},
		onCancel: () => {
			isUploading.value = false
		},
	})
}
</script>
```

## What Belongs In The Provider

Put rendering concerns in the notification provider.

- toast placement
- colors and icons
- progress UI
- close behavior

Keep domain wording and timing decisions in the feature that triggered the notification.

## Why This Is Separate

Notifications stay decoupled so that other parts of Ginjou can use them without hardwiring your UI stack. That matters for undoable mutations, global feedback, and cross-cutting flows such as auth or realtime status messages.

## Practical Guidance

- Use success and error notifications for user-visible action outcomes.
- Use progress only for long-running or cancelable flows.
- Keep notification text short and specific to the action that just happened.
- Let the provider own presentation, and let the feature own meaning.
