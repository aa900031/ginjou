Data access in Ginjou starts with a fetcher. Query and mutation composables call the fetcher, cache results with TanStack Query, and optionally connect that flow to notifications, auth-aware error handling, and realtime subscriptions.

## Mental Model

Every data flow in Ginjou follows the same path.

1. A component calls a query or mutation composable.
2. The composable resolves the active fetcher.
3. The fetcher translates the generic request into backend-specific HTTP or SDK calls.
4. TanStack Query stores the result and manages refetching, invalidation, and status.

That separation is what keeps the framework portable. Your UI talks to a stable Ginjou API even when the backend changes.

## Fetcher Contract

The fetcher is the backend contract used by all data composables.

```ts
interface Fetcher {
	getList?: (props: GetListProps) => Promise<GetListResult>
	getMany?: (props: GetManyProps) => Promise<GetManyResult>
	getOne?: (props: GetOneProps) => Promise<GetOneResult>
	createOne?: (props: CreateOneProps) => Promise<CreateResult>
	createMany?: (props: CreateManyProps) => Promise<CreateManyResult>
	updateOne?: (props: UpdateOneProps) => Promise<UpdateResult>
	updateMany?: (props: UpdateManyProps) => Promise<UpdateManyResult>
	deleteOne?: (props: DeleteOneProps) => Promise<DeleteOneResult>
	deleteMany?: (props: DeleteManyProps) => Promise<DeleteManyResult>
	custom?: (props: CustomProps) => Promise<CustomResult>
}
```

You do not need to implement every method on day one. Add only the operations your application uses.

## Root Setup

Register the query client and one or more fetchers at the root of the app.

::code-group

```vue [Vue]
<script setup lang="ts">
import { defineFetchersContext, defineQueryClientContext } from '@ginjou/vue'
import { createFetcher } from '@ginjou/with-rest-api'
import { QueryClient } from '@tanstack/vue-query'

defineQueryClientContext(new QueryClient())

defineFetchersContext({
	default: createFetcher({
		url: 'https://api.example.com',
	}),
	analytics: createFetcher({
		url: 'https://analytics.example.com',
	}),
})
</script>

<template>
	<RouterView />
</template>
```

::

The rest of the app does not import those fetchers directly. It asks Ginjou for data by resource name or by explicit `fetcherName`.

## Query Composables

Use low-level query composables when you want direct control over what is fetched and when.

| Composable | Use It For | Notes |
| :--- | :--- | :--- |
| `useGetList` | Paginated or filtered collections | The main low-level list query |
| `useGetOne` | One record by ID | Good for detail pages or edit forms |
| `useGetMany` | Many records by ID | Useful when one query returns related IDs |
| `useGetInfiniteList` | Load-more and infinite scroll | Keeps results page-shaped |
| `useCustom` | Query-shaped endpoints outside the CRUD surface | Good for reports or backend-specific endpoints |

### Shared Request Primitives

The same request primitives are reused across the query layer.

- `filters`: field, operator, and value tuples that the provider maps to backend query syntax.
- `sorters`: ordered field and direction pairs.
- `pagination`: page-based state such as `current` and `perPage`, or provider-specific pagination details.
- `meta`: the provider-specific escape hatch for backend features that do not belong in the generic contract.

```ts
const { records } = useGetList({
	resource: 'products',
	filters: [
		{ field: 'status', operator: 'eq', value: 'active' },
		{ field: 'title', operator: 'contains', value: 'desk' },
	],
	sorters: [
		{ field: 'created_at', order: 'desc' },
	],
	pagination: {
		current: 1,
		perPage: 20,
	},
	meta: {
		select: 'id,title,price',
	},
})
```

### useGetList

Use `useGetList` to fetch a collection with optional pagination, filters, sorters, and backend-specific `meta`.

::code-group

```vue [Vue]
<script setup lang="ts">
import { ref } from 'vue'
import { useGetList } from '@ginjou/vue'

const status = ref<'draft' | 'published'>('published')

const { records, data, isFetching, error } = useGetList({
	resource: 'posts',
	pagination: {
		current: 1,
		perPage: 10,
	},
	filters: [
		{ field: 'status', operator: 'eq', value: status },
	],
	sorters: [
		{ field: 'created_at', order: 'desc' },
	],
})
</script>

<template>
	<div v-if="isFetching">
		Loading posts...
	</div>
	<div v-else-if="error">
		Failed to load posts.
	</div>
	<div v-else>
		<p>Total: {{ data?.total }}</p>
		<ul>
			<li v-for="post in records" :key="post.id">
				{{ post.title }}
			</li>
		</ul>
	</div>
</template>
```

::

`records` gives you the current array of items. `data?.total` gives you the server-reported total count.

### useGetOne

Use `useGetOne` when the page needs one specific record.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useGetOne } from '@ginjou/vue'

const { record, isFetching, error } = useGetOne({
	resource: 'posts',
	id: '42',
})
</script>

<template>
	<div v-if="isFetching">
		Loading...
	</div>
	<div v-else-if="error">
		Could not load the record.
	</div>
	<article v-else-if="record">
		<h1>{{ record.title }}</h1>
		<p>{{ record.content }}</p>
	</article>
</template>
```

::

### useGetInfiniteList

Use `useGetInfiniteList` when the UI loads one page at a time.

::code-group

```vue [Vue]
<script setup lang="ts">
import { computed } from 'vue'
import { useGetInfiniteList } from '@ginjou/vue'

const { records, hasNextPage, fetchNextPage, isFetching } = useGetInfiniteList({
	resource: 'posts',
	pagination: {
		perPage: 20,
	},
})

const flatRecords = computed(() => records.value?.flat() ?? [])
</script>

<template>
	<ul>
		<li v-for="post in flatRecords" :key="post.id">
			{{ post.title }}
		</li>
	</ul>
	<button v-if="hasNextPage" :disabled="isFetching" @click="fetchNextPage()">
		Load more
	</button>
</template>
```

::

The raw `records` value is nested by page. Flatten it only when the UI needs one continuous list.

### useCustom

Use `useCustom` when the request is still a query, but it does not fit one of the standard list, many, or one-record shapes.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useCustom } from '@ginjou/vue'

const { record, isFetching, error } = useCustom({
	url: '/reports/monthly',
	method: 'get',
	query: {
		month: '2025-01',
	},
})
</script>

<template>
	<div v-if="isFetching">
		Loading report...
	</div>
	<div v-else-if="error">
		Could not load the report.
	</div>
	<pre v-else>{{ record }}</pre>
</template>
```

::

Reach for `useCustom` when the backend endpoint is query-like, but not naturally modeled as `getOne`, `getMany`, or `getList`.

## Mutation Composables

Mutations follow the same pattern: choose the resource, then call `mutate` or `mutateAsync` with the payload. These composables also expose pending state and cooperate with the rest of the data layer, including auth-aware error handling, notifications, realtime publishing, and invalidation behavior where that mutation type supports it.

| Composable | Use It For | Notes |
| :--- | :--- | :--- |
| `useCreateOne` | Create one record | The base write path for create pages |
| `useCreateMany` | Create many records at once | Good for imports or bulk creation |
| `useUpdateOne` | Update one record | The base write path for edit pages |
| `useUpdateMany` | Update many records at once | Useful for batch status changes |
| `useDeleteOne` | Delete one record | Supports undoable flows when notifications are available |
| `useDeleteMany` | Delete many records | Useful for batch destructive actions |
| `useCustomMutation` | Non-standard write endpoints | Good for workflow steps or server-side actions |

### useCreateOne

::code-group

```vue [Vue]
<script setup lang="ts">
import { reactive } from 'vue'
import { useCreateOne } from '@ginjou/vue'

const formData = reactive({
	title: '',
	status: 'draft',
})

const { mutateAsync: createPost, isPending } = useCreateOne({
	resource: 'posts',
})

async function submit() {
	await createPost({
		params: {
			title: formData.title,
			status: formData.status,
		},
	})
}
</script>
```

::

### useDeleteOne

`useDeleteOne` follows the same shape as the other mutation composables. It is the low-level building block behind delete buttons, confirm dialogs, and undoable delete flows.

## Case Study: Admin Products List

An admin products page usually needs more than one concern at the same time:

- fetch the current page of records
- apply fixed business filters such as active tenant or store
- let the user search and sort
- display loading, empty, and error states

Start with `useGetList` when the page already owns its own state, or move up to `useList` when you want one controller to manage the full page flow.

## Case Study: Combining Standard And Custom Queries

It is common for one screen to read standard resources and one backend-specific endpoint together.

```vue
<script setup lang="ts">
import { useCustom, useGetOne } from '@ginjou/vue'

const order = useGetOne({
	resource: 'orders',
	id: 'order-42',
})

const shipmentTimeline = useCustom({
	url: '/shipments/order-42/timeline',
	method: 'get',
	fetcherName: 'warehouse',
})
</script>
```

The surrounding page still speaks the same Ginjou language. Only the fetcher translation changes.

### useCustomMutation

Use `useCustomMutation` when the backend flow is mutation-like, but does not fit one of the standard create, update, or delete operations.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useCustomMutation } from '@ginjou/vue'

const exportReport = useCustomMutation()

async function submit() {
	await exportReport.mutateAsync({
		url: '/reports/export',
		method: 'post',
		payload: {
			month: '2025-01',
			format: 'csv',
		},
	})
}
</script>
```

::

Keep `useCustomMutation` for exceptional mutation endpoints. If the backend uses these shapes everywhere, write a custom fetcher instead of repeating custom calls in components.

## Error Handling

Ginjou does not normalize every backend error into one application-wide error shape. The `error` value you receive from a query or mutation is still provider-specific.

- network and validation details usually stay in the original backend error
- auth-aware failures are still checked through `useCheckError`
- user-facing messages can be centralized through notification hooks and notify callbacks

That means the right pattern is usually:

1. let the provider preserve the real backend error
2. let Ginjou handle auth failures centrally
3. handle feature-specific validation or business errors close to the page

## Meta

`meta` is the escape hatch for provider-specific behavior.

The exact `meta` shape depends on the provider.

- See [Directus Provider](/backend/directus) for `meta.query`.
- See [Supabase Provider](/backend/supabase) for `meta.select`, `meta.count`, and `meta.idColumnName`.

Use `meta` for backend-specific switches. Use a custom fetcher when the entire request model needs translation.

## Multiple Fetchers

Multiple fetchers are registered by name in `defineFetchersContext`. Those names are then used by composables through the `fetcherName` prop.

The important rule is simple:

1. Register each backend under a stable name.
2. Pass `fetcherName` when a low-level composable should use a non-default backend.
3. Use resource metadata only when route-aware controllers should infer that backend automatically.

::code-group

```vue [Vue]
<script setup lang="ts">
import { defineFetchersContext, defineQueryClientContext, useGetOne } from '@ginjou/vue'
import { createFetcher as createRestFetcher } from '@ginjou/with-rest-api'
import { QueryClient } from '@tanstack/vue-query'

defineQueryClientContext(new QueryClient())

defineFetchersContext({
	default: createRestFetcher({
		url: 'https://api.example.com',
	}),
	analytics: createRestFetcher({
		url: 'https://analytics.example.com',
	}),
})

const { record } = useGetOne({
	resource: 'reports',
	id: '42',
	fetcherName: 'analytics',
})
</script>
```

::

For low-level composables such as `useGetOne`, `useGetList`, `useCustom`, or `useCustomMutation`, `fetcherName` is the direct way to choose the backend.

If you are using route-aware controllers such as `useShow`, `useEdit`, or `useList`, put `meta.fetcherName` on the resource definition instead. Those controllers can infer the correct backend from the resolved resource.

## Advanced Example

This is a common pattern for a custom dashboard or embedded admin widget that mixes one query with a couple of low-level mutations.

::code-group

```vue [Vue]
<script setup lang="ts">
import { ref } from 'vue'
import { useCreateOne, useGetList, useUpdateOne } from '@ginjou/vue'

const title = ref('')

const posts = useGetList({
	resource: 'posts',
	pagination: { current: 1, perPage: 5 },
	sorters: [{ field: 'id', order: 'desc' }],
})

const createPost = useCreateOne({ resource: 'posts' })
const publishPost = useUpdateOne({ resource: 'posts' })

async function addDraft() {
	await createPost.mutateAsync({
		params: { title: title.value, status: 'draft' },
	})
	title.value = ''
}

async function publish(id: string | number) {
	await publishPost.mutateAsync({
		resource: 'posts',
		id,
		params: { status: 'published' },
	})
}
</script>
```

::

This is the point where Ginjou is most useful: you can build a focused feature with only the pieces you need, without first adopting a page controller.

## Choose The Right Level

- Use query and mutation composables when the page is unusual, embedded, or highly customized.
- Use controllers when the page follows a standard CRUD route.
- Use a custom fetcher when backend translation is the main problem.

That division is what keeps Ginjou flexible without becoming vague.
