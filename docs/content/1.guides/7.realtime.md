Realtime support in Ginjou is optional and provider-driven. When you register a realtime provider, query and mutation composables can subscribe to resource events and publish changes back to the system.

## Mental Model

Realtime in Ginjou is built around event transport, not one specific backend.

1. The realtime provider knows how to subscribe and optionally publish.
2. Queries and controllers can opt into realtime behavior.
3. `auto` mode handles standard invalidation for you.
4. `manual` mode lets the page decide how to react.

That split is important because some pages only need automatic refetching, while others need conflict handling or feature-specific event logic.

## Realtime Contract

```ts
interface Realtime {
	subscribe: (props: SubscribeProps) => UnsubscribeFn
	unsubscribe?: (props: UnsubscribeProps) => void
	publish?: (event: RealtimeEvent) => void
}
```

`subscribe` is required. `unsubscribe` and `publish` are optional.

## Setup

::code-group

```vue [Vue]
<script setup lang="ts">
import { defineRealtimeContext } from '@ginjou/vue'

defineRealtimeContext({
	subscribe({ channel, callback }) {
		const socket = connectToChannel(channel)
		socket.on('message', callback)

		return () => {
			socket.off('message', callback)
			socket.close()
		}
	},
	publish(event) {
		console.log('publish event', event)
	},
})
</script>
```

## A Practical Starting Point

For most applications, start with one realtime provider and let ordinary queries run in automatic mode. That gives you freshness without making every page understand event transport.

## Realtime Modes

Queries and controllers can work in three modes.

| Mode | Behavior |
| :--- | :--- |
| `off` | No subscription is created. |
| `auto` | Incoming events invalidate the relevant query scope automatically. |
| `manual` | You receive the event and decide what to do yourself. |

`auto` is the default choice for standard admin flows. `manual` is useful when a page needs custom logic such as conditional patching, inline conflict handling, or selective refetching.

## Auto Mode

In auto mode, a query subscribes to the relevant resource channel and invalidates itself when a matching event arrives.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useGetOne } from '@ginjou/vue'

const { record } = useGetOne({
	resource: 'posts',
	id: '42',
})
</script>
```

### Typical Use Case

Auto mode is the right default for:

- admin detail pages
- standard lists that should stay fresh
- dashboards where refetching is enough

If the UI does not need to inspect the incoming event payload, stay in auto mode.

## Manual Mode

In manual mode, Ginjou gives you the event and leaves the response strategy to your code.

::code-group

```vue [Vue]
<script setup lang="ts">
import { RealtimeMode } from '@ginjou/core'
import { useGetOne } from '@ginjou/vue'

const query = useGetOne({
	resource: 'posts',
	id: '42',
	realtime: {
		mode: RealtimeMode.Manual,
		callback(event) {
			if (event.action === 'updated') {
				query.refetch()
			}
		},
	},
})
</script>
```

## Query Integration

The data layer already knows how to subscribe based on the current query shape.

- `useGetList` subscribes at the resource level.
- `useGetOne` subscribes at the resource level with record context.
- `useGetMany` subscribes with the requested ID set.

On the mutation side, create, update, and delete composables can publish `created`, `updated`, and `deleted` events when the provider implements `publish`.

That means most CRUD pages can benefit from realtime without introducing any extra feature code.

## useSubscribe

Use `useSubscribe` when a component needs a channel that is not already modeled by a data query.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useSubscribe } from '@ginjou/vue'

useSubscribe({
	channel: 'resources/posts',
	actions: ['created', 'updated', 'deleted'],
	callback(event) {
		console.log('event', event)
	},
})
</script>
```

::code-group

```vue [Vue]
<script setup lang="ts">
import { useSubscribe } from '@ginjou/vue'

const subscription = useSubscribe({
	channel: 'presence/editors',
	actions: ['joined', 'left'],
	callback(event) {
		console.log('presence event', event)
	},
})

function stopWatchingPresence() {
	subscription.stop()
```

## usePublish

Use `usePublish` when you want to send a custom event.

::code-group

```vue [Vue]
<script setup lang="ts">
import { usePublish } from '@ginjou/vue'

const publish = usePublish()

function announceCursorMove() {
	publish({
		channel: 'resources/posts',
		action: 'cursor-moved',
		payload: { id: '42' },
	})
}
</script>
```

## Practical Guidance

- Start with `auto` mode.
- Use `manual` when the page needs a custom merge or invalidation strategy.
- Keep custom event shapes local to the feature that owns them.
- Let queries own data freshness first. Add custom channels only when the feature genuinely needs them.
