Realtime support in Ginjou is optional and provider-driven. In practice, it is used mainly through data composables: query composables subscribe to backend events, and mutation composables can publish events after successful changes.

## Mental Model

Realtime in Ginjou is built around event transport, not one specific backend.

1. The realtime provider defines how subscriptions and publishes work.
2. Data composables opt into realtime through their `realtime` option.
3. Query composables usually subscribe.
4. Mutation composables can publish.
5. `useSubscribe` and `usePublish` are available when a feature needs a channel outside the data layer.

That split is important because most CRUD screens only need automatic freshness, while a smaller number of features need custom event logic.

## Realtime Contract

```ts
interface Realtime {
	subscribe: (props: SubscribeProps) => UnsubscribeKey
	unsubscribe?: (key: UnsubscribeKey) => void
	publish?: (event: RealtimeEvent) => void
	options?: {
		mode?: RealtimeModeValue
		callback?: SubscribeCallbackFn
		params?: Record<string, unknown>
	}
}
```

`subscribe` is required. `unsubscribe`, `publish`, and `options` are optional.

Context-level `options` provide defaults for realtime-enabled data composables. Per-call `realtime` props are merged on top of them.

## Setup

::code-group

```vue [Vue]
<script setup lang="ts">
import { RealtimeMode } from '@ginjou/core'
import { defineRealtimeContext } from '@ginjou/vue'

defineRealtimeContext({
	subscribe({ channel, callback }) {
		socketManager.subscribe(channel, callback)
		return channel
	},
	unsubscribe(key) {
		socketManager.unsubscribe(key)
	},
	publish(event) {
		console.log('publish event', event)
	},
	options: {
		mode: RealtimeMode.Auto,
		params: {
			tenant: 'team-1',
		},
	},
})
</script>
```

## With Data Composables

Realtime is designed to be attached to the data layer first.

Query composables that subscribe:

- `useGetList`
- `useGetInfiniteList`
- `useGetOne`
- `useGetMany`

Mutation composables that can publish on success when the provider implements `publish`:

- `useCreateOne`
- `useCreateMany`
- `useUpdateOne`
- `useUpdateMany`
- `useDeleteOne`
- `useDeleteMany`
- `useCustomMutation`

Controllers built from these composables inherit the same behavior indirectly. For example, route-aware list or edit controllers benefit because they are composed on top of realtime-aware data hooks.

## Auto Mode

In auto mode, a query subscribes to the relevant channel and invalidates the matching resource scope when an event arrives.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useGetList } from '@ginjou/vue'

const { records } = useGetList({
	resource: 'posts',
```

## Manual Mode

In manual mode, Ginjou gives you the event and leaves the response strategy to your code.

::code-group

```vue [Vue]
<script setup lang="ts">
import { RealtimeMode } from '@ginjou/core'
import { useGetOne } from '@ginjou/vue'

const query = useGetOne({
	resource: 'posts',
	id: '42',
	realtime: {
		mode: RealtimeMode.Manual,
		callback(event) {
			if (event.action === 'updated') {
				query.refetch()
			}
		},
	},
})
</script>
```

## Custom Channel And Params

When the provider needs a different channel name or provider-specific subscription params, set them on the data composable.

::code-group

```vue [Vue]

const query = useGetOne({
	resource: 'posts',
	id: '42',
	realtime: {
		channel: 'tenant/posts',
		params: {
			tenant: 'team-1',
```

## Options

Data composables use a `realtime` option to control how they interact with the provider.

### `mode`

| Mode | Behavior |
| :--- | :--- |
| `off` | No subscription is created. |
| `auto` | Incoming events invalidate the relevant resource query scope automatically. This is the default. |
| `manual` | Ginjou does not auto-invalidate. Your callback decides how to react. |

Use `auto` for standard CRUD screens. Use `manual` when the page needs custom merge, refetch, or warning logic.

### `channel`

`channel` overrides the default subscription channel.

- If omitted, Ginjou uses `resources/${resource}`.
- Use it when the backend channel naming scheme does not match the default resource path.

### `callback`

`callback` receives the incoming realtime event.

- In `auto` mode, Ginjou invalidates first and then calls your callback.
- In `manual` mode, the callback is where your feature decides what to do.

### `params`

`params` carries provider-specific subscription data.

- Use it when the backend transport expects extra subscription metadata.
- Context-level `options.params` and per-call `realtime.params` are merged together.

## useSubscribe

Use `useSubscribe` when a component needs a channel that is not already modeled by a data query.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useSubscribe } from '@ginjou/vue'

useSubscribe({
	channel: 'resources/posts',
	actions: ['created', 'updated', 'deleted'],
	callback(event) {
		console.log('event', event)
	},
})
</script>
```

## usePublish

Use `usePublish` when you want to send a custom event.

::code-group

```vue [Vue]
<script setup lang="ts">
import { usePublish } from '@ginjou/vue'

const publish = usePublish()

function announceCursorMove() {
	publish({
		channel: 'resources/posts',
		action: 'cursor-moved',
		payload: { id: '42' },
	})
}
</script>
```

## Practical Guidance

- Start by wiring the provider, then let data composables own freshness.
- Use `auto` first. Reach for `manual` only when the page needs custom event handling.
- Override `channel` and `params` only when the backend transport really needs it.
- Use `useSubscribe` and `usePublish` for feature-level channels that are not naturally part of the data layer.
