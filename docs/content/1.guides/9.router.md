Ginjou has a router contract because route-aware resources and controllers need a small, stable navigation surface. It is not a replacement for the full router API of your framework.

## What The Router Contract Is For

Ginjou only needs a few capabilities from a router:

- navigate to a path
- replace instead of push when needed
- go back
- resolve a route object into a string path
- read the current location and react when it changes

Everything else, such as guards, named routes, middleware, scroll behavior, and route transitions, still belongs to the framework router.

## Router Contract

```ts
interface Router {
	go: RouterGoFn<any>
	back: RouterBackFn
	resolve: RouterResolveFn<any>
	getLocation: RouterGetLocationFn<any>
	onChangeLocation: RouterOnChangeLocationFn<any>
}
```

The low-level route object that Ginjou works with is equally small.

```ts
interface RouterGoParams<TMeta = unknown> {
	to?: string
	type?: 'push' | 'replace'
	query?: Record<string, string | number | null | undefined>
	hash?: string
	keepHash?: boolean
	keepQuery?: boolean
	meta?: TMeta
}
```

Register the router once near the app root.

```vue
<script setup lang="ts">
import { defineRouterContext } from '@ginjou/vue'

defineRouterContext(customRouterAdapter)
</script>
```

## useLocation

`useLocation` exposes the current route in Ginjou's router shape.

It returns a ref with these main fields:

- `path`
- `params`
- `query`
- `hash`
- `meta`

```vue
<script setup lang="ts">
import { computed } from 'vue'
import { useLocation } from '@ginjou/vue'

const location = useLocation()
const currentOrderId = computed(() => location.value?.params?.id)
</script>
```

Use this when the component needs low-level route state. If the page needs the active Ginjou resource instead, use [Resources](/guides/resources) through `useResource`.

## useGo And useBack

`useGo` is the low-level navigation function. `useBack` mirrors the browser or router back action.

```vue
<script setup lang="ts">
import { useBack, useGo } from '@ginjou/vue'

const go = useGo()
const back = useBack()

function openOrders() {
	go({ to: '/admin/orders', query: { status: 'pending' } })
}
</script>
```

Use these composables when the destination is already known as a path or when the app is doing something more custom than a resource action.

## Resource-Aware Navigation With useNavigateTo

`useNavigateTo` is the high-level navigation helper that connects the router contract to resource definitions.

```vue
<script setup lang="ts">
import { useNavigateTo } from '@ginjou/vue'

const navigateTo = useNavigateTo()

function openOrder(orderId: string) {
	navigateTo({
		resource: 'admin-orders',
		action: 'show',
		id: orderId,
	})
}
</script>
```

This is why resources matter. `useNavigateTo` looks up the resource definition, resolves the requested action, builds the path, and then calls the low-level router `go()` function.

Use `useNavigateTo` for list, create, show, and edit navigation. Use `useGo` when you need a lower-level path change.

## Building URLs With useResolvePath

`useResolvePath` turns low-level `RouterGoParams` into a string URL.

```vue
<script setup lang="ts">
import { computed } from 'vue'
import { useResolvePath } from '@ginjou/vue'

const resolvePath = useResolvePath()
const exportHref = computed(() => resolvePath({
	to: '/admin/orders',
	query: { format: 'csv' },
}))
</script>
```

Use it when the UI needs a final href for a normal link, download link, or preview element.

## Where Resources Enter The Picture

Resources and router abstractions solve different problems.

- the router contract moves around the app
- resources describe which path belongs to which resource action
- `useNavigateTo` uses both
- `useResource` reads both

That is why `syncRoute` for lists belongs in [List](/guides/list): it is a page-level concern built on top of this lower-level router contract.

## Use The Framework Router For Framework Concerns

Ginjou's router contract exists for Ginjou's internal needs. When the application needs full router features, prefer the framework router directly.

- use the framework router for guards and middleware
- use the framework router for named routes and advanced route records
- use the framework router for transition hooks and scroll behavior
- use Ginjou's router contract for resource-aware controllers and navigation helpers
