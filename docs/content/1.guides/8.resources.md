title: Resources
description: Define route-aware resources and map them to actions, fetchers, and navigation.

Resources are how Ginjou understands your application structure. They tell the framework which URLs belong to which resource, which action is active, and which fetcher should be used. Use them when a page needs route-aware controllers, one shared route-to-action map, resource hierarchy through `meta.parent`, navigation visibility through `meta.hide`, or backend selection through `meta.fetcherName`.

## Mental Model

Resources are the routing map for the rest of Ginjou.

1. You define resource names and route patterns.
2. Ginjou matches the current location against those definitions.
3. Controllers and composables use the resolved resource to know which action and record are active.
4. Resource metadata adds hierarchy, visibility, and fetcher selection.

Without resources, Ginjou can still run low-level queries. With resources, route-aware controllers become possible.

## Resource Definition

```ts
interface ResourceDefinition {
	name: string
	list?: string | ResourceActionParse
	create?: string | ResourceActionParse
	edit?: string | ResourceActionParse
	show?: string | ResourceActionParse
	meta?: {
		parent?: string
		hide?: boolean
		fetcherName?: string
		[key: string]: any
	}
}
```

The core metadata fields used by Ginjou are:

- `parent` for hierarchy
- `hide` for navigation control
- `fetcherName` for multi-backend setups

## Why Resources Matter Early

Resources are not only for large admin shells. They become useful as soon as the app needs any of these patterns:

- `useList`, `useCreate`, `useEdit`, or `useShow` should infer the current resource from the route
- navigation should target list, create, show, or edit without hard-coding paths everywhere
- different areas of the application should resolve to different fetchers without repeating that choice in every query

## Setup

Register resources at the root.

::code-group

```vue [Vue]
<script setup lang="ts">
import { defineResourceContext } from '@ginjou/vue'

defineResourceContext({
	resources: [
		{
			name: 'posts',
			list: '/posts',
			create: '/posts/new',
			edit: '/posts/:id/edit',
			show: '/posts/:id',
		},
	],
})
</script>
```

::

Once resources exist, controllers such as `useList`, `useEdit`, and `useShow` can resolve the current route into a resource-aware context.

## Standard Resource Patterns

Most CRUD applications can start with one predictable resource map.

```ts
const resources = {
	resources: [
		{
			name: 'posts',
			list: '/posts',
			create: '/posts/new',
			show: '/posts/:id',
			edit: '/posts/:id/edit',
		},
		{
			name: 'users',
			list: '/users',
			show: '/users/:id',
		},
	],
}
```

For string-based `show` and `edit` patterns, Ginjou expects the dynamic parameter to be named `id`.

::tip
If your route uses another key such as `:uuid`, use a custom parser instead of a string `show` or `edit` pattern.
::

## One App Can Have Front Office And Back Office Resources

It is normal for one project to expose different resources for storefront and admin pages even when they point at related backend entities.

```ts
defineResourceContext({
	resources: [
		{
			name: 'store-products',
			list: '/products',
			show: '/products/:id',
			meta: {
				fetcherName: 'catalog',
			},
		},
		{
			name: 'admin-products',
			list: '/admin/products',
			create: '/admin/products/new',
			show: '/admin/products/:id',
			edit: '/admin/products/:id/edit',
			meta: {
				fetcherName: 'admin',
			},
		},
	],
})
```

This keeps page intent explicit. The storefront and the admin are not forced to share the same resource name only because they happen to work with products.

## Hierarchies, Visibility, And Fetcher Selection

Use `meta.parent` to express nested structures.

```ts
{
	name: 'post-comments',
	list: '/posts/:postId/comments',
	show: '/posts/:postId/comments/:id',
	meta: {
		parent: 'posts',
	},
}
```

Use `meta.hide` when a resource should remain resolvable but should not appear in generated navigation.

```ts
{
	name: 'post-revisions',
	list: '/posts/:postId/revisions',
	meta: {
		hide: true,
	},
}
```

Use `meta.fetcherName` when different resources should go to different backends.

```ts
{
	name: 'analytics',
	list: '/analytics',
	meta: {
		fetcherName: 'warehouse',
	},
}
```

That selection is picked up automatically by the data composables and controllers.

## Custom Parsing

If a route does not fit a simple string pattern, use a custom parser.

```ts
const resources = [
	{
		name: 'posts',
		show: {
			pattern: '/posts/:uuid',
			parse(location) {
				const uuid = location.path.replace('/posts/', '')

				if (/^[0-9a-f-]{36}$/.test(uuid)) {
					return {
						action: 'show',
						id: uuid,
					}
				}
			},
		},
	},
]
```

Use this sparingly. String patterns cover most applications and are much easier to scan.

## useResource

`useResource` resolves the active resource from the current location, or resolves a named resource directly when you pass `name`.

::code-group

```vue [Vue]
<script setup lang="ts">
import { computed } from 'vue'
import { useResource } from '@ginjou/vue'

const resolved = useResource()

const title = computed(() => {
	if (!resolved.value)
		return 'Unknown page'

	if (resolved.value.action === 'edit')
		return `Edit ${resolved.value.resource.name}`

	if (resolved.value.action === 'show')
		return `View ${resolved.value.resource.name}`

	return resolved.value.resource.name
})
</script>
```

::

`useResource` is the bridge between the current route and the rest of the CRUD layer. Read [Router](/guides/router) next if navigation should target resource actions instead of hard-coded paths.
