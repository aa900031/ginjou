Resources are how Ginjou understands your application structure. They tell the framework which URLs belong to which resource, which action is active, and which fetcher should be used. Use them when a page needs route-aware controllers, one shared route-to-action map, resource hierarchy through `meta.parent`, navigation visibility through `meta.hide`, or backend selection through `meta.fetcherName`. If a feature is only an isolated widget, low-level queries may still be enough on their own.

## Mental Model

Resources are the routing map for the rest of Ginjou.

1. You define resource names and route patterns.
2. Ginjou matches the current location against those definitions.
3. Controllers and composables use the resolved resource to know which action and record are active.
4. Resource metadata adds hierarchy, visibility, and fetcher selection.

Without resources, Ginjou can still run low-level queries. With resources, route-aware controllers become possible.

## Resource Definition

```ts
interface ResourceDefinition {
	name: string
	list?: string | ResourceActionParse
	create?: string | ResourceActionParse
	edit?: string | ResourceActionParse
	show?: string | ResourceActionParse
	meta?: {
		parent?: string
		hide?: boolean
		fetcherName?: string
		[key: string]: any
	}
}
```

The core metadata fields used by Ginjou are:

- `parent` for hierarchy
- `hide` for navigation control
- `fetcherName` for multi-backend setups

## Setup

Register resources at the root.

::code-group

```vue [Vue]
<script setup lang="ts">
import { defineResourceContext } from '@ginjou/vue'

defineResourceContext({
	resources: [
		{
			name: 'posts',
			list: '/posts',
			create: '/posts/new',
			edit: '/posts/:id/edit',
			show: '/posts/:id',
		},
	],
})
</script>
```

## Standard Resource Patterns

Most CRUD applications can start with one predictable resource map.

```ts
const resources = {
	resources: [
		{
			name: 'posts',
			list: '/posts',
			create: '/posts/new',
			show: '/posts/:id',
			edit: '/posts/:id/edit',
		},
		{
			name: 'users',
			list: '/users',
			show: '/users/:id',
		},
	],
}
```

Once this is registered, route-aware controllers no longer need each page to repeat the same resource wiring.

These are the common route mappings.

| Action | Example |
| :--- | :--- |
| `list` | `/posts` |
| `create` | `/posts/new` |
| `show` | `/posts/:id` |
| `edit` | `/posts/:id/edit` |

For string-based `show` and `edit` patterns, Ginjou expects the dynamic parameter to be named `id`.

::tip
If your route uses another key such as `:uuid`, use a custom parser instead of a string `show` or `edit` pattern.
::

### Common Mistakes

- using a string `show` or `edit` pattern without an `:id` segment
- treating `meta.hide` as access control instead of navigation visibility
- putting backend translation into page code instead of `meta.fetcherName` or the fetcher itself

## Nest Resources

Use `meta.parent` to express nested structures.

```ts
const resources = [
	{
		name: 'posts',
		list: '/posts',
		show: '/posts/:id',
	},
	{
		name: 'post-comments',
		list: '/posts/:postId/comments',
		show: '/posts/:postId/comments/:id',
		meta: {
			parent: 'posts',
		},
	},
]
```

This is useful for breadcrumbs, nested menus, and any page logic that needs to understand resource hierarchy.

## Hidden Resources

Use `meta.hide` when a resource should remain resolvable but should not appear in generated navigation.

```ts
{
	name: 'post-revisions',
	list: '/posts/:postId/revisions',
	meta: {
		hide: true,
	},
}
```

## Multiple Fetchers

Use `meta.fetcherName` when different resources should go to different backends.

```ts
const resources = [
	{
		name: 'posts',
		list: '/posts',
		meta: {
			fetcherName: 'content',
		},
	},
	{
		name: 'analytics',
		list: '/analytics',
		meta: {
			fetcherName: 'warehouse',
		},
	},
]
```

That selection is picked up automatically by the data composables and controllers.

## Custom Parsing

If a route does not fit a simple string pattern, use a custom parser.

```ts
const resources = [
	{
		name: 'posts',
		show: {
			pattern: '/posts/:uuid',
			parse(location) {
				const uuid = location.path.replace('/posts/', '')

				if (/^[0-9a-f-]{36}$/.test(uuid)) {
					return {
						action: 'show',
						id: uuid,
					}
				}
			},
		},
	},
]
```

Use this sparingly. String patterns cover most applications and are much easier to scan.

## useResource

`useResource` resolves the active resource from the current location, or resolves a named resource directly when you pass `name`.

- It can expose the resolved resource definition.
- It can expose the parsed action from the current route.
- It can expose the current record ID when the route pattern includes one.

This is useful for breadcrumbs, page titles, layout logic, and route-aware UI that should adapt to the active resource.

::code-group

```vue [Vue]
<script setup lang="ts">
import { computed } from 'vue'
import { useResource } from '@ginjou/vue'

const resolved = useResource()
const posts = useResource({ name: 'posts' })

const title = computed(() => {
	if (!resolved.value)
		return 'Unknown page'

	if (resolved.value.action === 'edit')
		return `Edit ${resolved.value.resource.name}`

	if (resolved.value.action === 'show')
		return `View ${resolved.value.resource.name}`

	return resolved.value.resource.name
})
</script>
```
