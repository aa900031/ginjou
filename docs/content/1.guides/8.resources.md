Resources are how Ginjou understands your application structure. They tell the framework which URLs belong to which resource, which action is active, and which fetcher should be used.

## Mental Model

Resources are the routing map for the rest of Ginjou.

1. You define resource names and route patterns.
2. Ginjou matches the current location against those definitions.
3. Controllers and composables use the resolved resource to know which action and record are active.
4. Resource metadata adds hierarchy, visibility, and fetcher selection.

Without resources, Ginjou can still run low-level queries. With resources, route-aware controllers become possible.

## Quick Reference

Use resources when the page needs any of these:

- route-aware controllers such as `useList`, `useEdit`, or `useShow`
- one stable place to map URLs to resource actions
- backend selection through `meta.fetcherName`
- hierarchy or navigation metadata such as `parent` and `hide`

If the feature is an isolated widget or a non-route-bound panel, low-level queries may be enough on their own.

## Resource Definition

```ts
interface ResourceDefinition {
	name: string
	list?: string | ResourceActionParse
	create?: string | ResourceActionParse
	edit?: string | ResourceActionParse
	show?: string | ResourceActionParse
	meta?: {
		parent?: string
		hide?: boolean
		fetcherName?: string
		[key: string]: any
	}
}
```

The core metadata fields used by Ginjou are:

- `parent` for hierarchy
- `hide` for navigation control
- `fetcherName` for multi-backend setups

## Setup

Register resources at the root.

::code-group

```vue [Vue]
<script setup lang="ts">
import { defineResourceContext } from '@ginjou/vue'

defineResourceContext({
	resources: [
		{
			name: 'posts',
			list: '/posts',
			create: '/posts/new',
			edit: '/posts/:id/edit',
			show: '/posts/:id',
		},
	],
})
</script>
```

## A Standard Resource Set

This is a good default for a CRUD application.

```ts
const resources = {
	resources: [
		{
			name: 'posts',
			list: '/posts',
			create: '/posts/new',
			show: '/posts/:id',
			edit: '/posts/:id/edit',
		},
		{
			name: 'users',
			list: '/users',
			show: '/users/:id',
		},
	],
}
```

Once this is registered, route-aware controllers no longer need each page to repeat the same resource wiring.

## Standard Route Patterns

These are the common mappings.

| Action | Example |
| :--- | :--- |
| `list` | `/posts` |
| `create` | `/posts/new` |
| `show` | `/posts/:id` |
| `edit` | `/posts/:id/edit` |

### Resolved Resource Example

::code-group

```vue [Vue]
<script setup lang="ts">
import { computed } from 'vue'
import { useResource } from '@ginjou/vue'

const resolved = useResource()

const title = computed(() => {
	if (!resolved.value)
		return 'Unknown page'

	if (resolved.value.action === 'edit')
		return `Edit ${resolved.value.resource.name}`

	if (resolved.value.action === 'show')
		return `View ${resolved.value.resource.name}`

	return resolved.value.resource.name
})
</script>
```

### Important Rule For String Patterns

For string-based `show` and `edit` patterns, the dynamic parameter must be named `id`.

```ts
{
	name: 'posts',
	show: '/posts/:id',
	edit: '/posts/:id/edit',
}
```

If your route uses another key such as `:uuid`, use a custom parser instead.

### Common Mistakes

- using a string `show` or `edit` pattern without an `:id` segment
- treating `meta.hide` as access control instead of navigation visibility
- putting backend translation into page code instead of `meta.fetcherName` or the fetcher itself

## Hierarchical Resources

Use `meta.parent` to express nested structures.

```ts
const resources = [
	{
		name: 'posts',
		list: '/posts',
		show: '/posts/:id',
	},
	{
		name: 'post-comments',
		list: '/posts/:postId/comments',
		show: '/posts/:postId/comments/:id',
		meta: {
			parent: 'posts',
		},
	},
]
```

This is useful for breadcrumbs, nested menus, and any page logic that needs to understand resource hierarchy.

## Hidden Resources

Use `meta.hide` when a resource should remain resolvable but should not appear in generated navigation.

```ts
{
	name: 'post-revisions',
	list: '/posts/:postId/revisions',
	meta: {
		hide: true,
	},
}
```

## Multiple Fetchers

Use `meta.fetcherName` when different resources should go to different backends.

```ts
const resources = [
	{
		name: 'posts',
		list: '/posts',
		meta: {
			fetcherName: 'content',
		},
	},
	{
		name: 'analytics',
		list: '/analytics',
		meta: {
			fetcherName: 'warehouse',
		},
	},
]
```

That selection is picked up automatically by the data composables and controllers.

### Cross-Backend Example

```ts
const resources = {
	resources: [
		{
			name: 'posts',
			list: '/posts',
			meta: {
				fetcherName: 'content',
			},
		},
		{
			name: 'analytics',
			list: '/analytics',
			meta: {
				fetcherName: 'warehouse',
			},
		},
	],
}
```

That lets one application keep a single resource map even when the data comes from different systems.

## Custom Parsing

If a route does not fit a simple string pattern, use a custom parser.

```ts
const resources = [
	{
		name: 'posts',
		show: {
			pattern: '/posts/:uuid',
			parse(location) {
				const uuid = location.path.replace('/posts/', '')

				if (/^[0-9a-f-]{36}$/.test(uuid)) {
					return {
						action: 'show',
						id: uuid,
					}
				}
			},
		},
	},
]
```

Use this sparingly. String patterns cover most applications and are much easier to scan.

## Common Resource Shapes

These patterns cover most applications.

| Need | Recommended shape |
| :--- | :--- |
| Standard CRUD pages | `list`, `create`, `show`, `edit` string patterns |
| Nested records | child resource with `meta.parent` |
| Hidden utility routes | normal resource with `meta.hide: true` |
| Different backend ownership | `meta.fetcherName` |
| Non-standard route parameter | custom parser |

## A Good Workflow

When adding a new resource, use this order.

1. Define the route patterns.
2. Decide whether the resource belongs to a parent.
3. Decide whether it should be visible in navigation.
4. Decide which fetcher should own it.
5. Only then add controllers or pages on top.

## A Good Default

Keep resource names stable, keep action patterns predictable, and only introduce custom parsing when the route model genuinely needs it. Clear resources make every controller easier to understand.
