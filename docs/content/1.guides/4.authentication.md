Authentication in Ginjou is provider-driven. You register an auth contract once, then use composables to log in, log out, check the current session, inspect the current identity, and react to backend auth errors.

## Mental Model

Authentication in Ginjou is split into a few focused jobs.

1. The auth provider knows how to talk to the backend.
2. `useLogin` and `useLogout` run auth mutations.
3. `useAuthenticated` answers whether the current session is valid.
4. `useGetIdentity` loads display data for the signed-in user.
5. `useCheckError` decides whether backend errors should invalidate the session.

That split matters because login state and identity data are related, but they are not the same concern.

## Auth Contract

```ts
interface Auth {
	login: (params?: any) => Promise<LoginResult | void>
	logout: (params?: any) => Promise<LogoutResult | void>
	check: (params?: any) => Promise<CheckAuthResult>
	checkError: (error: any) => Promise<CheckAuthErrorResult>
	getIdentity?: (params?: any) => Promise<GetIdentityResult>
}
```

Only `getIdentity` is optional.

## Setup

Register the auth provider at the root.

::code-group

```vue [Vue]
<script setup lang="ts">
import { defineAuthContext } from '@ginjou/vue'
import { createAuth } from '@ginjou/with-supabase'

defineAuthContext(createAuth({ client }))
</script>
```

::

Provider-specific login types live in the backend packages. For example:

- Supabase supports `password`, `oauth`, `idtoken`, `otp`, `sso`, and `otp-token`.
- Directus supports `password` and `sso`.

## Typical Placement

These composables usually live in different parts of the app.

- The sign-in page calls `useLogin`.
- The protected shell or route boundary calls `useAuthenticated`.
- The header, account menu, or profile area calls `useGetIdentity`.
- Built-in data queries and mutations already route request failures through `useCheckError`.

That separation keeps session checks, identity display, and auth mutations from being mixed into the same component.

## useLogin

`useLogin` wraps `auth.login` with mutation state and success handling.

### Composition

- Action: `auth.login`
- Mutation on success: invalidates auth-related query state
- Composable on success: redirects according to the router integration

That means `useLogin` is usually enough for the page-level flow. You only need custom logic when the backend response requires extra branching.

### Example

::code-group

```vue [Vue]
<script setup lang="ts">
import { reactive, ref } from 'vue'
import { useLogin } from '@ginjou/vue'

const credentials = reactive({
	email: '',
	password: '',
})
const errorMessage = ref('')

const { mutateAsync: login, isPending } = useLogin()

async function submit() {
	errorMessage.value = ''

	try {
		await login({
			type: 'password',
			params: {
				email: credentials.email,
				password: credentials.password,
			},
		})
	}
	catch {
		errorMessage.value = 'Invalid email or password.'
	}
}
</script>

<template>
	<form class="stack" @submit.prevent="submit">
		<input v-model="credentials.email" type="email" placeholder="Email">
		<input v-model="credentials.password" type="password" placeholder="Password">
		<p v-if="errorMessage">{{ errorMessage }}</p>
		<button type="submit" :disabled="isPending">
			{{ isPending ? 'Signing in...' : 'Sign in' }}
		</button>
	</form>
</template>
```

::

## useLogout

`useLogout` wraps `auth.logout`.

### Composition

- Action: `auth.logout`
- Mutation on success: invalidates query state unless logout is configured to skip invalidation
- Composable on success: redirects to the resolved destination, which defaults to `/login`
- Mutation on error: shows an auth logout notification and can redirect if the error payload provides a redirect target

`useLogout` uses the same page-level pattern as `useLogin`, but its success path is broader because logout usually clears cached application state and returns the user to a public screen.

### Example

::code-group

```vue [Vue]
<script setup lang="ts">
import { useLogout } from '@ginjou/vue'

const { mutateAsync: logout } = useLogout()
</script>

<template>
	<button @click="logout()">
		Sign out
	</button>
</template>
```

::

## useAuthenticated

`useAuthenticated` checks the current session state with `auth.check`.

### Composition

- Action: `auth.check`
- Query key: auth check state plus optional params
- Query enabled rule: runs only when the provider exposes `check`, and respects `queryOptions.enabled`
- Query callbacks: forwards success and error callbacks from `queryOptions`

Use this at route boundaries, shells, or guards. It is a query because session validity is state that may need to be re-read, not a one-time mutation.

### Example

Use it in guards, shell layouts, or route-level components that need to know whether the user is currently authenticated.

::code-group

```vue [Vue]
<script setup lang="ts">
import { computed } from 'vue'
import { useAuthenticated } from '@ginjou/vue'

const { data, isLoading } = useAuthenticated()
const isAuthenticated = computed(() => data.value?.authenticated === true)
</script>

<template>
	<div v-if="isLoading">
		Checking session...
	</div>
	<RouterView v-else-if="isAuthenticated" />
	<div v-else>
		Please sign in to continue.
	</div>
</template>
```

::

This belongs in a shell, route layout, or page boundary. It keeps session checks centralized instead of scattering them across every action button.

## useGetIdentity

`useGetIdentity` calls `auth.getIdentity` when the provider implements it.

### Composition

- Action: `auth.getIdentity`
- Query key: identity state plus optional params
- Query enabled rule: runs only when the provider exposes `getIdentity`, and respects `queryOptions.enabled`
- Query callbacks: forwards success and error callbacks from `queryOptions`

This is intentionally separate from `useAuthenticated`. A session can be valid even when the UI does not need profile data yet.

### Example

This is typically used for profile menus, avatars, and headers.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useGetIdentity, useLogout } from '@ginjou/vue'

const { data: identity, isLoading } = useGetIdentity<{
	name: string
	email: string
	avatarUrl?: string
}>()
const { mutateAsync: logout } = useLogout()
</script>

<template>
	<header class="toolbar">
		<div v-if="isLoading">Loading profile...</div>
		<div v-else-if="identity">
			<img v-if="identity.avatarUrl" :src="identity.avatarUrl" alt="">
			<div>
				<strong>{{ identity.name }}</strong>
				<p>{{ identity.email }}</p>
			</div>
		</div>
		<button @click="logout()">Sign out</button>
	</header>
</template>
```

::

Keep this separate from `useAuthenticated`. One answers whether the session is valid. The other provides user-facing profile data.

## useCheckError

`useCheckError` lets the auth provider decide whether a backend error is really an auth failure.

In normal app code, you often do not need to call it manually. Built-in data composables such as `useGetList`, `useGetOne`, `useCreateOne`, `useUpdateOne`, `useDeleteOne`, `useCustom`, and `useCustomMutation` already create `useCheckError` internally and pass it into their error handlers.

That means auth-aware request handling is usually automatic. The built-in integration covers the main query and mutation surface, including:

- `useGetList`, `useGetInfiniteList`, `useGetOne`, and `useGetMany`
- `useCreateOne`, `useCreateMany`, `useUpdateOne`, and `useUpdateMany`
- `useDeleteOne` and `useDeleteMany`
- `useCustom` and `useCustomMutation`

Controllers inherit that behavior indirectly because they are built on top of those data composables.

Call `useCheckError` manually only when you are handling a backend error outside the built-in data layer.

::code-group

```vue [Vue]
<script setup lang="ts">
import { useCheckError } from '@ginjou/vue'

const { mutateAsync: checkError } = useCheckError()

async function handleError(error: unknown) {
	await checkError(error)
}
</script>
```

::

## Session Invalidation Flow

What counts as an auth failure depends on the provider, but the flow is the same.

1. A query or mutation fails.
2. The built-in Ginjou hook forwards that error to `auth.checkError`.
3. The provider decides whether the error means "session is no longer valid" or "this is still a normal request error."
4. If the provider returns `logout: true`, Ginjou runs the logout flow.
5. If not, the error stays a normal request error for the current screen.

This is why validation errors, business rule failures, and permission errors should usually stay regular request errors. `checkError` should stay focused on session invalidation.

## Provider-Specific Flows Live In Backend Pages

Authentication in this guide stays at the framework level.

- Read [Supabase](/backend/supabase) for supported login types and session behavior.
- Read [Directus](/backend/directus) for Directus-specific auth capabilities.
- Read [Custom](/backend/custom) when your auth flow is attached to a custom fetcher or a non-standard backend.

## Practical Guidance

- Put backend-specific login parameters in the provider docs, not in generic app code.
- Use `useAuthenticated` for session presence and `useGetIdentity` for display data.
- Keep `checkError` strict. It should only force logout for real session failures.
- Start with one protected shell and one login page. Add finer auth UX only after that baseline is stable.
