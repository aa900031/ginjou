Ginjou has a small i18n contract because system messages, notification text, and resource labels still need translation. It is not meant to replace the full i18n feature set of your framework.

## What Ginjou's I18n Layer Is For

Use Ginjou's i18n layer for the parts of the system that Ginjou itself emits or consumes.

- success and error notifications
- auth and data-layer system messages
- resource labels when the app wants translated resource-aware copy
- a shared current locale when translated system messages should react to locale changes

For full application copy, dates, pluralization, number formatting, and page-level content, prefer the framework i18n solution directly.

## I18n Contract

```ts
interface I18n {
	translate: (key: string, params?: Record<string, any>) => string | undefined
	getLocale?: () => string
	setLocale?: (locale: string, options?: unknown) => void | Promise<void>
	onChangeLocale?: (handler: (locale: string) => void) => () => void
}
```

Only `translate` is required. Locale management is optional.

## Setup

Register the i18n contract near the app root.

```vue
<script setup lang="ts">
import { defineI18nContext } from '@ginjou/vue'
import { createI18n } from '@ginjou/with-vue-i18n'

defineI18nContext(createI18n())
</script>
```

If you already use another i18n library, write a small adapter that implements the same contract.

## useTranslate

`useTranslate` returns the current translation function in Ginjou's lightweight shape.

```vue
<script setup lang="ts">
import { useNotify, useTranslate } from '@ginjou/vue'

const notify = useNotify()
const translate = useTranslate()

function announceSave() {
	notify({
		type: 'success',
		message: translate('notifications.saveSuccess'),
	})
}
</script>
```

Use `useTranslate` when the message belongs to the system layer that Ginjou participates in.

## useLocale

`useLocale` returns a writable locale ref, but only when the registered i18n provider also implements `getLocale`, `setLocale`, and `onChangeLocale`.

```vue
<script setup lang="ts">
import { useLocale } from '@ginjou/vue'

const locale = useLocale()
</script>

<template>
	<select v-model="locale">
		<option value="en">English</option>
		<option value="ja">Japanese</option>
	</select>
</template>
```

If the provider only implements `translate`, `useLocale` will throw. That is expected: locale management is optional in the contract.

## What To Translate Through Ginjou

Good fits for the Ginjou i18n layer:

- generic success and error notifications
- undoable action copy
- auth-related system messages
- resource labels that appear inside generic CRUD pages

Less suitable fits:

- large marketing pages or product copy
- complex ICU messages and advanced pluralization rules
- date and number formatting policies that belong to the app UI layer

## Use The Framework I18n Tool For App-Level Concerns

Ginjou's i18n layer is intentionally small. When the application needs the full power of i18n, keep using the framework-native tool.

- use `vue-i18n` or the Nuxt i18n stack for page content and advanced formatting
- bridge that framework tool into Ginjou so system messages stay aligned
- avoid duplicating translation ownership across two different systems

That boundary is the same design choice as routing: Ginjou only asks for the minimal contract it needs.
