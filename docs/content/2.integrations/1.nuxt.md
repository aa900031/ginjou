`@ginjou/nuxt` adds Nuxt-specific integration on top of `@ginjou/vue`. The programming model stays the same, but the module provides router integration, query hydration, and async composables for SSR-friendly route pages.

## What The Nuxt Module Adds

The Nuxt module changes the setup surface in three important ways.

1. Providers are still registered at the root app level.
2. Router integration is supplied by the module, so you normally do not call `defineRouterContext` yourself.
3. Async composables participate in Nuxt server rendering and hydrate into the client query cache.

Choose this integration when route-level data should be ready during SSR or hydration.

## Installation

The example below uses the REST API provider. Replace it with the backend package your application actually uses.

::code-group
---
sync: package-manager
---

```bash [pnpm]
pnpm add @ginjou/nuxt @ginjou/vue @tanstack/vue-query @ginjou/with-rest-api
```

```bash [yarn]
yarn add @ginjou/nuxt @ginjou/vue @tanstack/vue-query @ginjou/with-rest-api
```

```bash [npm]
npm install @ginjou/nuxt @ginjou/vue @tanstack/vue-query @ginjou/with-rest-api
```

## Setup

### nuxt.config.ts

Register the module first.

```ts [nuxt.config.ts]
export default defineNuxtConfig({
	modules: ['@ginjou/nuxt'],
})
```

### app.vue

Register Ginjou providers in `app.vue` or `app/app.vue`.

::code-group

```vue [Nuxt]
<script setup lang="ts">
import { defineFetchersContext, defineQueryClientContext, defineResourceContext } from '@ginjou/vue'
import { createFetcher } from '@ginjou/with-rest-api'
import { QueryClient } from '@tanstack/vue-query'

defineQueryClientContext(new QueryClient())

defineFetchersContext({
	default: createFetcher({
		url: 'https://api.example.com',
	}),
})

defineResourceContext({
	resources: [
		{
			name: 'posts',
			list: '/posts',
			show: '/posts/:id',
		},
	],
})
</script>

<template>
	<NuxtLayout>
		<NuxtPage />
	</NuxtLayout>
</template>
```

::

You do not need to call `defineRouterContext` in Nuxt. The module provides router integration for you.

Keep this registration unconditional. In Nuxt, conditional provider setup is an easy way to create SSR or hydration mismatches.

## Auto Imports

The module auto-imports the common page-facing composables from `@ginjou/vue`, plus the Nuxt async wrappers.

Auto-imported groups include:

- low-level query and mutation hooks such as `useGetList`, `useGetOne`, `useCreateOne`, `useUpdateOne`, `useDeleteOne`, `useCustom`, and `useCustomMutation`
- controllers such as `useCreate`, `useEdit`, `useList`, `useShow`, `useSelect`, and `useInfiniteList`
- auth and authz hooks such as `useLogin`, `useLogout`, `useAuthenticated`, `useGetIdentity`, `useCheckError`, `usePermissions`, and `useCanAccess`
- resource, notification, and realtime helpers such as `useNavigateTo`, `useResource`, `useResourcePath`, `useNotify`, `usePublish`, and `useSubscribe`

These low-level router and i18n composables are not auto-imported by the module:

- `useGo`
- `useBack`
- `useLocation`
- `useResolvePath`
- `useTranslate`
- `useLocale`

If you need them, import them from `@ginjou/vue` explicitly. The same is true for the root `define*Context` registration functions.

## Async Composables

The Nuxt module provides SSR-ready async wrappers for the main query and controller flows.

| Area | Async composables |
| :--- | :--- |
| Query | `useAsyncGetOne`, `useAsyncGetMany`, `useAsyncGetList`, `useAsyncGetInfiniteList` |
| Controllers | `useAsyncShow`, `useAsyncList`, `useAsyncEdit`, `useAsyncSelect`, `useAsyncInfiniteList` |
| Auth and authz | `useAsyncAuthenticated`, `useAsyncGetIdentity`, `useAsyncPermissions`, `useAsyncCanAccess` |

Use these in route-level pages when the data should be present during server rendering.

These are the complete async wrappers provided by the module today. There is no `useAsyncCreate`, so create and update flows still use the standard mutation composables on the client side.

::code-group

```vue [Nuxt]
<script setup lang="ts">
const { records, isLoading } = await useAsyncGetList({
	resource: 'posts',
})
</script>

<template>
	<div v-if="isLoading">
		Loading...
	</div>
	<ul v-else>
		<li v-for="post in records" :key="post.id">
			{{ post.title }}
		</li>
	</ul>
</template>
```

::

The async composables keep the same result shape as the normal Vue composables. The difference is that they participate in Nuxt's async data flow, so the server-rendered payload hydrates into the client query cache.

## Practical Guidance

- Put all provider registration in the root app component.
- Keep registration unconditional to avoid SSR hydration mismatches.
- Use async composables for route-level data that should be ready during server rendering.
- Use the standard Vue composables for purely client-side interactions.
