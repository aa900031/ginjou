`@ginjou/vue` is the base Vue adapter for Ginjou.

It exposes Ginjou's core contracts through Vue provide/inject, adds TanStack Query-backed data hooks, and includes the controller layer for resource-oriented pages.

`@ginjou/nuxt` builds on the same primitives, so this package is the foundation for both plain Vue apps and the Nuxt integration.

## Installation

Install `@ginjou/vue` together with `@tanstack/vue-query`.

Ginjou uses TanStack Query for query and mutation state in Vue, so both packages are part of the base setup.

::code-group
---
sync: package-manager
---

```bash [pnpm]
pnpm add @ginjou/vue @tanstack/vue-query
```

```bash [yarn]
yarn add @ginjou/vue @tanstack/vue-query
```

```bash [npm]
npm install @ginjou/vue @tanstack/vue-query
```

```bash [bun]
bun add @ginjou/vue @tanstack/vue-query
```

::

## Setup

The smallest useful Vue setup starts with two contexts:

1. `defineQueryClientContext()` for TanStack Query state.
2. `defineFetchersContext()` for data access.

Register them once near the app root so every Ginjou composable reads the same shared contracts.

::code-group

```vue [App.vue]
<script setup lang="ts">
import { defineFetcher } from '@ginjou/core'
import { defineFetchersContext, defineQueryClientContext } from '@ginjou/vue'
import { QueryClient } from '@tanstack/vue-query'

defineQueryClientContext(new QueryClient({
	defaultOptions: {
		queries: {
			staleTime: 5 * 60 * 1000,
		},
	},
}))

defineFetchersContext({
	default: defineFetcher({
		async getList() {
			return {
				data: [],
				total: 0,
			}
		},
		async getOne({ id }) {
			return {
				data: { id },
			}
		},
	}),
})
</script>
```

::

From there, add router, auth, authz, notifications, i18n, realtime, and resources only when the app needs them.

## Contexts

`@ginjou/vue` keeps all shared contracts explicit through Vue provide/inject.

Each `defineXContext()` call installs one app-level contract. Hooks, controllers, and lower-level helpers read from those contracts later.

| Provider | When to add it | Guide |
| --- | --- | --- |
| `defineQueryClientContext` | Always for the data layer | [Data](/guides/data) |
| `defineFetchersContext` | Always for the data layer | [Data](/guides/data) |
| `defineAuthContext` | When the app supports login, logout, or identity | [Authentication](/guides/authentication) |
| `defineAuthzContext` | When the app checks permissions or access rules | [Authorization](/guides/authorization) |
| `defineRouterContext` | When navigation, current location, or controller route sync matter | [Router](/guides/router) |
| `defineI18nContext` | When the app translates text or switches locale | [I18n](/guides/i18n) |
| `defineNotificationContext` | When the app shows toast-like feedback | [Notifications](/guides/notifications) |
| `defineRealtimeContext` | When queries subscribe or mutations publish realtime events | [Realtime](/guides/realtime) |
| `defineResourceContext` | When the app defines shared resource metadata | [Resources](/guides/resources) |

In practice, most Vue apps start with query client and fetchers, then add the rest only when the app grows into those concerns.

This page only covers the integration shape. Use the linked guides for the full contract and behavior of each context.
