`@ginjou/vue` is the base frontend integration for Ginjou. It provides the context registration functions, the low-level query and mutation composables, and the higher-level controllers built on top of them.

Use this package when you are building a Vue application directly. If you are using Nuxt, read [Nuxt](/integrations/nuxt) instead.

## What This Package Does

The Vue integration keeps Ginjou explicit.

1. Register providers once near the root.
2. Add the optional capabilities your application needs: router, auth, authz, notifications, realtime, and i18n.
3. Build pages with low-level composables first.
4. Use controllers when the route already fits a standard CRUD shape.

There is no monolithic plugin. Ginjou works by composing small providers around the parts of the app that need them.

## Setup

Register Ginjou in `App.vue` so the rest of the component tree can resolve the shared providers.

::code-group

```vue [Vue]
<script setup lang="ts">
import { defineFetchersContext, defineQueryClientContext, defineResourceContext, defineRouterContext } from '@ginjou/vue'
import { createFetcher } from '@ginjou/with-rest-api'
import { createRouter } from '@ginjou/with-vue-router'
import { QueryClient } from '@tanstack/vue-query'

defineQueryClientContext(new QueryClient())

defineFetchersContext({
	default: createFetcher({
		url: 'https://api.example.com',
	}),
})

defineResourceContext({
	resources: [
		{
			name: 'posts',
			list: '/posts',
			show: '/posts/:id',
			edit: '/posts/:id/edit',
		},
	],
})

defineRouterContext(createRouter())
</script>

<template>
	<RouterView />
</template>
```

::

This example assumes Vue Router is already installed in the application entry, so `createRouter()` can access the current router instance from inside `App.vue`.

## Root Providers

These are the root registration functions exposed by `@ginjou/vue`.

| Function | Purpose |
| :--- | :--- |
| `defineQueryClientContext` | Register the TanStack Query client. |
| `defineFetchersContext` | Register one or more fetchers. |
| `defineResourceContext` | Register resources and route patterns. |
| `defineRouterContext` | Register router integration. |
| `defineAuthContext` | Register authentication. |
| `defineAuthzContext` | Register authorization. |
| `defineNotificationContext` | Register notifications. |
| `defineRealtimeContext` | Register realtime support. |
| `defineI18nContext` | Register translations. |

Register only what the application needs. For example, pages built entirely with low-level data hooks do not require resources or router integration yet.

## Working Style In Vue

The recommended structure is:

1. `App.vue` registers providers.
2. Resources define route-aware CRUD pages.
3. Route components call low-level composables or high-level controllers.

That keeps framework wiring centralized and feature logic local.

## First Route-Aware Page

A practical first milestone is one resource and one list page.

1. Register a resource such as `products` in `App.vue`.
2. Create a Vue Router page component for `/products`.
3. Call `useList({ resource: 'products' })` inside that page.
4. Render `records` and one navigation action, such as opening the show page.

Once that works, the application has the full route-aware path in place: router -> resource -> controller -> fetcher.

## Choosing Composables Or Controllers

Controllers such as `useList`, `useCreate`, `useEdit`, and `useShow` are convenience compositions. They are built from the same lower-level hooks you can call directly.

Use controllers when the page matches a standard CRUD route. Use low-level composables when the screen is more specialized.

## Recommended Supporting Packages

- `@ginjou/with-vue-router` for router integration
- `@ginjou/with-vue-i18n` for translation integration
- one backend provider such as `@ginjou/with-rest-api`, `@ginjou/with-supabase`, or `@ginjou/with-directus`
