# I18n Reference

Use this reference when the task includes adding localization, internationalization, or translation support to a Ginjou app.

## Provider Rule

I18n is an app-level integration. Register it once with `defineI18nContext` using the `@ginjou/with-vue-i18n` adapter.

## Setup

```typescript
import { defineI18nContext } from '@ginjou/vue'
import { createI18n } from '@ginjou/with-vue-i18n'
import { createI18n as createVueI18n } from 'vue-i18n'

const i18n = createVueI18n({ locale: 'en', messages: { en: { ... } } })
defineI18nContext(createI18n(i18n))
```

## Provider Contract

```typescript
interface I18n {
  translate:       (key: string, params?: Record<string, any>) => string | undefined
  getLocale?:      () => string
  setLocale?:      (locale: string, options?: any) => void | Promise<void>
  onChangeLocale?: (handler: (locale: string) => void) => () => void
}
```

## Composable Usage

```typescript
import { useTranslate } from '@ginjou/vue'

const { translate: t } = useTranslate()
t('resource.posts.name')  // returns translated string or undefined
```

## Guidance

- `@ginjou/with-vue-i18n` bridges `vue-i18n` to the Ginjou I18n interface.
- Ginjou uses translations internally for default notification messages and resource labels.
- You can register a custom adapter if you prefer a different i18n library.

## Common Mistakes

- Registering i18n conditionally or only in certain pages — register at the app root.
- Forgetting that `translate` may return `undefined` if a key is missing, not an error.
